The Five Agents
Each agent answers one question. No agent sees another's vote.
01
Sector Fit
Is this our kind of deal? Evaluates issuer type, state, and bond type against the firm's coverage map and strategic priorities.
02
Size & Capital
Can we afford this commitment? Compares par amount against capital limits, inventory position, and good faith deposit capacity.
03
Structure
Are the bidding rules workable? Purely technical—evaluates coupon constraints, basis of award, and maturity structure. No firm context needed.
04
Distribution
Can we sell these bonds? Assesses investor appetite by maturity, tax status, credit rating, and the firm's retail vs institutional strength.
05
Calendar
Do we have bandwidth? Checks analyst availability, pipeline capacity, and time to sale. The simplest agent—but its Pass is a hard veto.
—
Consensus
Deterministic Python. Not an LLM call. Any Pass at ≥80% confidence kills the deal. Unanimous interested proceeds. Everything else surfaces conditions for human review.

04
Deterministic Validation
Every LLM extraction goes through deterministic validation checks. These are pure math — no model involved. If a check fails, the extraction is retried with the specific error message.
Validation Engine
Retry mechanism: If any check fails, the system re-sends the original text to the LLM with the specific error message appended. This targeted feedback typically resolves extraction errors on the first retry.
05
Parallel Agent Screening
Five independent Claude API calls, each with a different system prompt and different NOS fields. No agent sees another's vote. This prevents anchoring bias and ensures each dimension is evaluated independently.
Why Multi-Agent Over Single LLM?
Single LLM: Hedges dealbreakers into caveats. If 4 dimensions look good and 1 is fatal, outputs "Conditional" with the problem buried in paragraph three.
Multi-Agent: Hard vetoes are enforced. Each agent has decomposed reasoning. Architecture mirrors real desk structure.
06
Deterministic Consensus
Pure Python. Not an LLM call. The consensus function applies four rules in priority order. The key design choice: any single high-confidence Pass kills the deal.

R2All five agents vote InterestedINTERESTED
R3Multiple low-confidence Pass votes (< 0.8)CONDITIONAL + Escalation
R4Mix of Interested and Conditional (no high-confidence Pass)CONDITIONAL

Key insight: The consensus function is where the MAS architecture pays off. A single LLM produces one blended judgment. The MAS produces five independent votes with five rationales, then a deterministic function enforces organizational decision rules that no LLM consistently respects.
