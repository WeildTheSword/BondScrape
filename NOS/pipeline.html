01
PDF Input
The pipeline begins with a Notice of Sale PDF. NOS documents have no standard format — they vary by financial advisor template, by state, and by issuer type. Documents range from 3 pages (a simple Texas MUD) to 15+ pages (a complex state authority). All contain the same core information: who is selling, what is being sold, and under what rules.

02
Text Extraction
pdftotext -layout is the only extraction method that works reliably across all NOS document types. Raw pdftotext drops maturity table amounts on PScript5 PDFs. pdfplumber produces run-together words. The -layout flag preserves the spatial alignment that makes maturity schedules parseable.

03
LLM Structured Extraction
The full extracted text is sent to Claude with a 55-feature JSON schema. The model reads the entire NOS in one context window and returns structured JSON matching the schema. Every field maps to a specific agent's input — the schema was designed so each agent reads only its relevant features.

04
Deterministic Validation
Every LLM extraction goes through deterministic validation checks. These are pure math — no model involved. If a check fails, the extraction is retried with the specific error message. This catches the most common LLM extraction errors: transposed numbers in maturity schedules, miscalculated deposits, and date logic failures.
Validation Engine
Retry mechanism: If any check fails, the system re-sends the original text to the LLM with the specific error message appended. Example: "Par amount sum ($2,925,000) does not match stated par amount ($2,930,000). Please re-extract the maturity schedule." This targeted error feedback typically resolves extraction errors on the first retry.
05
Parallel Agent Screening
Five independent Claude API calls, each with a different system prompt and different NOS fields. No agent sees another's vote. Each receives the validated NOS JSON (filtered to its relevant fields) plus the firm profile (filtered to its relevant context). This prevents anchoring bias and ensures each dimension is evaluated independently.
Why Multi-Agent Over Single LLM?
Single LLM: Hedges dealbreakers into caveats. If 4 dimensions look good and 1 is fatal, outputs "Conditional" with the problem buried in paragraph three.
Multi-Agent: Hard vetoes are enforced. Each agent has decomposed reasoning. Architecture mirrors real desk structure — different people with different data making different judgments.
06
Deterministic Consensus
Pure Python. Not an LLM call. The consensus function applies four rules in priority order. The key design choice: any single high-confidence Pass kills the deal. This is the behavior a single LLM consistently fails to enforce.
Consensus Function
R1 Any single Pass with confidence >= 0.8 PASS (Hard Veto)
R2 All five agents vote Interested INTERESTED
R3 Multiple low-confidence Pass votes (< 0.8) CONDITIONAL + Escalation
R4 Mix of Interested and Conditional (no high-confidence Pass) CONDITIONAL

Key insight: The consensus function is where the MAS architecture pays off. A single LLM produces one blended judgment. The MAS produces five independent votes with five rationales, then a deterministic function enforces organizational decision rules that no LLM consistently respects.
